## How this folder updates

This is a **living spec**. Edits land as the work converges:

- **Plan** (`01-plan.md`) — targeted edits, rarely wholesale rewrites. Each substantive change references the Round in `04` that drove it.
- **Tracking** (`05-tracking.md`) — checkboxes added as work lands. New work packages appended when discovered. The orchestrator owns checkbox writes during a run.
- **Research** (`02`/`03`) — mostly stable. Re-run `groundwork research` when a major new question lands; the action augments inside fences, hand-written prose outside is preserved.
- **Discussion** (`04`) — `groundwork review` appends "Round N" sections at the top (newest-first). Older sections retained for history.
- **Sub-plans** (`NN-*.md`) — created with `groundwork subplan`. Once a sub-plan's work lands, flip its `status` in `.groundwork.json.subplans` from `active` to `landed` (the file stays in place for history).
- **Artifact** (`artifact/index.html`) — living spec; edit by hand as the plan content evolves.
- **Board** (`artifact/board.html`) — regenerated by `groundwork refresh-board`; reads from the same `.groundwork.json` + `05-tracking.md` you've already updated.

## How to surface state from the artifact

The artifact publishes its current state (active tab, selection, layout) via `art.publishState`. From any terminal in Ikenga:

```bash
iyke state --json | jq '.shell.panes.leaves[].id'         # find the pane id
iyke iframe-state <pane> --json | jq '.state'             # read the published state
```

If you're an agent reading this README — that's how to check what's currently active before re-asking the user. Outside Ikenga, `art.publishState` is a no-op; the artifact still renders + works.
