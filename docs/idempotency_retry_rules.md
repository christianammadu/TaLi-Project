# SmartBookkeeper FOS Idempotency & Retry Rules

This document outlines the distributed safety, idempotency guarantees, and retry behaviors designed into the WhatsApp Financial Operating System (FOS) multi-agent architecture.

---

## 1. Core Principles

Our distributed multi-agent system is built around the **At-Least-Once Delivery** model. Under this model, network errors, processing drops, or webhook timeouts can cause messages/events to be retried. Downstream agents must handle duplicates gracefully without producing inconsistent states (double bookkeeping writes, duplicate inventory movements, or duplicate WhatsApp alerts).

---

## 2. Webhook / Intake Deduplication (AgentRouter)

All message traffic entering the system via the Meta WhatsApp API starts at the `AgentRouter`.

1. **Transaction Event Logging**: Incoming events are logged into the `webhook_events` table with the unique `whatsapp_message_id` as the primary lookup.
2. **Idempotency Guard**:
   - If the `status` of the `whatsapp_message_id` is `'received'`, `'processing'`, or `'processed'`, the router immediately terminates routing and returns `__DUPLICATE_DROP__`. This prevents duplicate processing of events that are already running or completed.
   - If the previous attempt has a `status` of `'failed'`, the router updates the status to `'processing'` and allows the event to be retried. This ensures that transient network failures or database locks can recover safely.

---

## 3. Ledger Agent Idempotency & Double-Entry Safeties

The `LedgerAgent` (Agent 2) is the single writer to the database.

1. **Event-Level Idempotency Check**:
   - Inside the database transaction boundary, the `LedgerAgent` verifies whether the unique `event_id` (a UUID generated by the `IntakeAgent`) already exists in the `processed_messages` table.
   - If it exists, the database transaction is immediately rolled back, and the agent returns a success confirmation (`Already processed (idempotency drop)`), bypassing redundant writes.
2. **Atomic SQL Transaction Bound**:
   - Every state-mutating command (e.g. creating a record, logging double-entry receivables/payables, logging inventory adjustments, or writing audit records) is wrapped inside a single atomic transaction block.
   - If any validation fails, or if a stock verification raises an insufficient stock flag requiring a user clarification prompt, the entire transaction is rolled back. No partial or orphaned state is committed.
   - On exception/failure, the `webhook_events` status is set to `'failed'`, allowing future retries from Meta to re-execute the intake pipeline.

---

## 4. CFO Agent Idempotency

The `CFOAgent` (Agent 3) generates formatted summaries and sends WhatsApp alerts back to the user.

1. **Notification Deduplication**:
   - The `CFOAgent` checks the `processed_messages` log before formatting alerts or compiling snapshots.
   - If the ledger update corresponding to the message ID was already processed and verified by the CFO, the alert is dropped (`Already processed`). This prevents sending duplicate or noisy WhatsApp notification cards to the customer.

---

## 5. Saga Error / Dead-Letter Compensation

1. **Ledger Failures**: When a database lock or commit failure is raised during a write session in the Ledger Agent, it:
   - Rolls back any active transactions.
   - Sets the `status` of the corresponding `webhook_events` row to `'failed'`.
   - Dispatches a failure payload to the `"ledger_errors"` channel.
2. **Compensation Routing**:
   - The `CFOAgent` listens to `"ledger_errors"`.
   - It intercepts the failure and formats a friendly recovery message for the user: *"⚠️ I understood your request, but our database encountered a temporary lock. Your transaction was not saved. Please try again."*
   - This ensures the client is always notified of a failure so they can re-initiate the command.
